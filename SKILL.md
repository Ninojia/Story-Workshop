# 故事工坊

一套专为企业内网文章打造的系统化写作工作流，支持从素材输入到终稿输出的全链路。

## 触发条件

**当用户表达以下意图时触发本 skill：**
- 写一篇内网文章 / KM文章
- 基于访谈记录写人物故事
- 写项目复盘 / 技术分享
- 写产品发布或功能介绍
- 基于奖项申报材料写人物故事（需要生成访谈提纲）
- 任何需要结构化产出的长文写作（>500字）

**典型触发语句：**
- "帮我写一篇关于XX的内网文章"
- "基于这份访谈记录写一篇文章"
- "帮我写个项目复盘"
- "我要写一篇技术分享"
- "把这个产品功能写成一篇介绍文章"
- "我有个奖项申报材料，想写成人物故事"

## 字数约束

**默认目标字数：约 2000 字。** 内网文章篇幅过长没人读，2000 字左右是最佳阅读体验区间（阅读时间约 5-7 分钟）。

- 用户未指定字数时，**一律按 2000 字**规划和写作
- 用户明确指定字数时，以用户要求为准
- 允许浮动范围：目标字数的 ±15%（即 1700-2300 字）
- 超过 2500 字需要用户明确确认，并说明理由

**各文章类型默认字数：**

| 文章类型 | 默认字数 | 说明 |
|---------|---------|------|
| 📖 人物/团队故事 | 2000 字 | 聚焦 1-2 个核心情节，不铺全时间线 |
| 📊 项目复盘 | 1800 字 | 聚焦 2-3 个关键决策，不面面俱到 |
| 🔧 技术分享 | 1800 字 | 聚焦核心方案，细节用代码/图辅助 |
| 🚀 产品介绍 | 1500 字 | 聚焦 3 个核心亮点，不贪多 |

## 核心理念

### 灵活流程，按需适配

本工坊提供两种工作模式，根据任务复杂度自动推荐（用户可随时切换）：

**🚀 快速模式** — 适合素材充分、目标明确的简单文章
```
📥 素材分析 → ✍️ 直接写作 → 🔍 快速审校
```

**🔬 完整模式** — 适合复杂文章、高质量要求、素材需要深度加工
```
📥 素材分析 → [📝 访谈提纲（素材不足时）] → 📋 文章规划 → ✍️ 内容执行 → 🔍 系统审校
```

### 模式选择判断标准

| 条件 | 推荐模式 |
|------|---------|
| 素材充足（访谈稿/文档已备齐），文章类型明确 | 🚀 快速 |
| 目标字数 ≤ 2000 字（默认情况） | 🚀 快速 |
| 用户明确说"快速出一版" | 🚀 快速 |
| 素材零散，需要深度梳理和取舍 | 🔬 完整 |
| 用户要求超过 2500 字 | 🔬 完整 |
| 文章类型为人物故事或深度复盘 | 🔬 完整 |
| 用户明确说"仔细打磨" | 🔬 完整 |

**重要：** 无论哪种模式，素材分析阶段不可跳过。这是产出质量的基石。

## 支持的素材类型

| 素材类型 | 说明 |
|---------|------|
| **本地文件**（.docx/.txt/.md/.pdf） | 访谈记录、项目文档、申报材料等 |
| **腾讯文档链接** | 通过腾讯文档 MCP 获取内容（需对应 MCP 配置） |
| **IMA 文档链接** | 通过 IMA 文档 MCP 获取内容（需对应 MCP 配置） |
| **口述需求** | 通过对话补全信息 |
| **混合素材** | 以上任意组合 |

## 支持的文章类型

### 📖 人物/团队故事
- 场景：基于访谈记录，讲述项目背后的人和事
- 风格：故事化叙事、场景感强、有情感温度
- 结构特征：以时间线或关键事件为骨架，穿插人物对话和细节
- 典型要素：开场钩子（关键场景切入）、人物弧光、冲突与转折、数据佐证、金句收尾

### 📊 项目复盘
- 场景：项目结束后的经验总结，面向团队或全公司分享
- 风格：条理清晰、干货密集、有数据支撑
- 结构特征：背景-挑战-方案-结果-反思 五段式
- 典型要素：关键决策点分析、数据对比（优化前后）、踩坑经验、可复用方法论

### 🔧 技术分享
- 场景：技术方案解读、架构设计分享、工程实践总结
- 风格：专业严谨但不枯燥，用类比让技术概念易懂
- 结构特征：问题驱动 → 方案选型 → 实现细节 → 效果验证
- 典型要素：技术背景铺垫、方案对比表格、核心代码/架构图说明、性能数据

### 🚀 产品发布/功能介绍
- 场景：新产品或新功能的内部/外部介绍
- 风格：轻松活泼、用户视角、突出价值
- 结构特征：痛点切入 → 产品方案 → 核心亮点 → 使用指引
- 典型要素：用户场景还原、功能亮点（3-5个）、效果对比、引导行动

## 各阶段概览

### 阶段1：素材分析
**Skill：** `story-workshop:素材分析`

**输入：** 访谈记录、文档、腾讯文档/IMA 文档链接、产品资料、或用户口述的需求

**产出：** 写作简报 → `docs/writing/YYYY-MM-DD-<topic>-brief.md`

**核心任务：**
- 理解素材，提取关键信息（人物、事件、数据、金句）
- 支持腾讯文档链接和 IMA 文档链接的自动获取（需对应 MCP）
- 判断文章类型，推荐写作风格
- 评估素材充分度，识别信息缺口
- 与用户确认写作方向
- 生成结构化写作简报

**流程结束后：**
- 素材不足 → 进入访谈提纲生成
- 快速模式 → 直接进入内容执行
- 完整模式 → 进入文章规划

### 阶段1.5：访谈提纲生成（素材不足时自动触发）
**Skill：** `story-workshop:访谈提纲生成`

**输入：**
已完成的写作简报 + 素材充分度评估

**产出：**
访谈提纲 → `docs/writing/YYYY-MM-DD-<topic>-interview-outline.md`

**触发条件（任一满足即触发）：**
- 文章类型为「人物/团队故事」但缺少人物直接引语
- 素材中只有事实/数据罗列，缺乏场景描述和细节（如奖项申报材料）
- 写作简报中「素材缺口」≥ 3 条
- 写作简报中「情感高光」和「素材金句」均为空

**核心任务：**
- 基于已有素材识别信息缺口（对话缺口、场景缺口、情感缺口等）
- 确定受访对象（当事人、上级、同事等）
- 为不同受访对象生成差异化的访谈问题
- 输出结构化的访谈提纲，方便 HR 直接使用

**流程结束后：**
- 用户完成访谈并提供记录 → 回到素材分析（补充分析）→ 更新写作简报 → 进入文章规划/内容执行
- 用户暂时无法访谈 → 基于现有素材继续，在写作中标注待补充信息

### 阶段2：文章规划（完整模式）
**Skill：** `story-workshop:文章规划`

**输入：** 已确认的写作简报

**产出：** 详细写作计划 → `docs/writing/YYYY-MM-DD-<topic>-plan.md`

**核心任务：**
- 设计文章结构（根据类型选择合适的骨架）
- 为每个章节明确：主题、要点、素材分配、字数目标
- 规划章节间的起承转合
- 识别需要补充的素材缺口

### 阶段3：内容执行
**Skill：**
- `story-workshop:内容执行`（批量模式）
- `story-workshop:子智能体写作`（迭代模式）

**输入：** 写作简报（快速模式）或写作计划（完整模式）

**产出：**
- 章节草稿 → `docs/writing/drafts/YYYY-MM-DD-<topic>/`
- 完整草稿 → `docs/writing/YYYY-MM-DD-<topic>-draft.md`

**核心任务：**
- 按计划逐节写作（或快速模式下一气呵成）
- 每节写完自检：风格一致性、逻辑通顺、衔接自然
- 设置人工检查点，避免大量返工

### 阶段4：文章审校
**Skill：** `story-workshop:文章审校`

**输入：** 完整草稿

**产出：**
- 审校报告 → `docs/writing/YYYY-MM-DD-<topic>-review.md`
- 终稿 → `docs/writing/YYYY-MM-DD-<topic>-final.md`

**核心任务（三轮审校）：**
1. **内容审校：** 逻辑通顺？论据充分？信息准确？
2. **表达审校：** 文字流畅？风格统一？节奏得当？
3. **规范审校：** 格式正确？用语规范？数据核实？

## 快速上手

### 用户操作流程

**快速模式：**
1. 提供素材（访谈稿/文档/腾讯文档链接/IMA 链接/口述需求）
2. 确认写作简报
3. 审阅草稿，提出修改意见
4. 确认终稿

**完整模式：**
1. 提供素材
2. 确认写作简报
3. [如素材不足] 确认访谈提纲 → 完成访谈 → 提供访谈记录
4. 确认文章大纲
5. 选择执行方式（批量 / 子智能体）
6. 在检查点审阅进度
7. 审阅终稿

### Agent 执行规则

1. **检测写作意图** — 用户提到"写文章"、"写一篇"、"帮我写"等关键词时触发
2. **优先获取素材** — 如果用户已提供文件（docx/pdf/txt/md）或链接（腾讯文档/IMA），主动读取和分析
3. **判断模式** — 根据素材充分度和任务复杂度推荐快速或完整模式
4. **素材不足时自动触发访谈提纲** — 检测到信息缺口时主动生成访谈提纲，不等用户要求
5. **不跳过素材分析** — 即使用户说"直接写"，也至少完成素材分析并生成简报
6. **中文输出** — 所有过程文档和最终文章均为中文
7. **尊重用户节奏** — 在关键节点等待用户确认，不自作主张推进

## 写作风格通则：「冷静的温度」

> 本风格体系基于对内外部优秀采写文章的系统性分析提炼而成——融合了外部顶级商业深度报道的叙事骨架与节奏感，以及内部优秀文化访谈的人情味与团队温度。

**核心理念：冷静的骨架，温暖的血肉。** 尊重读者智商，但关心文中的人。

无论哪种文章类型，均遵循以下准则：

### 准则一：开头一句话亮矛盾，不铺背景

第一句话必须包含一个具体的矛盾或张力，让读者想知道"然后呢"。矛盾要朝向"人"，而非纯宏观判断。

- ❌ "过去一年半，某项目获得技术突破奖……"（纯背景铺垫）
- ❌ "AI 的协作已经成为确定趋势。"（纯宏观判断，缺少人）
- ✅ "500 万条视频，每刷新一次模型要等半年——直到他们决定不再'造房子'，改'盖楼'。"

### 准则二：结构论点驱动，案例深挖撑腰

采用**「判断 → 案例 → 更大判断」**的螺旋上升结构，但案例要深挖到底——聚焦一个团队/一条线，不跳来跳去。

- 每一层判断都有**人 + 事 + 数据**三件套支撑，绝不空谈
- 小标题用**带判断的短句**（3-8 个字），不用纯引语、不用纯内容概括
- 示例：「不再造房子」比「技术架构升级」有力；「先跑起来」比「快速迭代策略」有温度

### 准则三：人物靠"做了什么"立住，引语是弹药

不贴标签（不写"他是一个追求极致的技术人"），不写心理活动（不写"他心里暗暗下定决心"）。

用一个具体动作让人物立体：
> "方案已经完美了——直到实际上线后的业务反馈，给了她当头一棒。"

引语使用规则：
- 保留口语感，但**删掉无效语气词**（"嗯""就是""然后"等）
- 每句引语必须携带信息或态度，不引用废话
- 引语是论证的"弹药"——每段关键论证都应有一句有力的直接引语支撑

### 准则四：数据嵌入叙事，回答"到底多大"

数据从不单独列表，而是嵌入在句子中，作为论证的"锤子"。

- ❌ 单独列数据清单
- ✅ "刷新时长从半年缩到一个月，年省 200 万——这不是优化，是换了条路。"
- 关键数据前后要有**对比感**（before vs after），对比才有冲击力
- 每个数据都应回答一个"到底有多大 / 多快 / 多少"的问题

### 准则五：比喻少而精，一个比喻管一段

不堆砌比喻。每个技术概念**最多配一个比喻**。

- 比喻选取**生活化场景**（亲切感），但表达要利落（克制感）
- ✅ "过去的做法像造房子搬家——而他们选择在原地盖楼。"
- ❌ "就像一个精巧的建筑师在搭建一座充满智慧的技术大厦……"

### 准则六：语气冷静但不冷漠

态度通过**选材和措辞**暗示，而非直说：

- 不用感叹号（极端情况最多一个）
- 不用"令人震惊""不可思议""非常伟大"等空洞情绪词
- "非常""极其""十分"全文出现不超过 3 次
- 选择报道什么、引用谁的话、强调哪个数据——这就是态度
- 允许在结尾处稍微升温，但靠事实和引语升温，不靠抒情

### 准则七：段落节奏——短长交替，3-5 句封顶

- 每段不超过 5 句话（中文约 150 字以内）
- 关键判断句**独立成段**（1-2 句），像呼吸的"换气点"
- 长句用逗号 / 破折号分层，但一句话只传递一个核心信息
- 段落间留白充分，阅读节奏舒适

### 准则八：金句藏在流动中，不刻意标粗

好的句子应该在阅读流中自然击中读者，不需要加粗、居中、单独装裱。

- 每 500-800 字至少有一个值得划线的句子（2000 字文章需 3-4 个金句）
- 金句放在段尾或独立成行，利用位置本身制造力量
- 如果一句话真的好，读者自己会停下来——不需要你帮他划重点

### 准则九：技术表达——说人话，但不装外行

- 专业术语**首次出现时用一句话解释**，之后直接用
- 不过度简化（读者不蠢），也不故弄玄虚
- 技术细节服务于"这件事为什么难、为什么值得做"的叙事，不为炫技
- 用生活化类比开路："小文件传输慢，就像送 100 瓶水，每次只装一瓶来回跑 100 趟"

### 准则十：结尾落在"未完成"上，留余味

- ❌ "相信在大家的努力下，未来一定会更好！"（口号式）
- ❌ "综上所述，本项目取得了以下成果……"（报告体）
- ✅ 落在一个未完成的动作、一句还在路上的话、一个正在发生的变化上
- 示例："先跑起来，就有了不断校准、验证的机会。"

### 反AI准则：让文章有毛边

AI 写作最大的破绽不是单句质量差，而是**全文太整齐**。以下是必须刻意打破的模式：

**1. 章节骨架不能重复**
- ❌ 每章都是"以前很费时 → AI自动化 → 省时间 → 去做更有价值的事"
- ✅ 不同章节有不同的叙事重心：有的重点在方法论，有的重点在态度，有的重点在失败，有的重点在工具链细节
- 检查方法：写完全文后，把每章骨架抽象成一句话。如果超过半数结构相似，必须重构

**2. 不是每个段落都要"完成"**
- ❌ 每章末尾都来一句升华金句（"这正是人和AI的默契""试错成本变低了，这可能是最大的礼物"）
- ✅ 有些章节就停在具体事实上，不做评论。留白比金句更有力
- 全文金句式总结句**不超过2处**，其余章节用事实或引语自然收束

**3. 注入"人类犹豫感"**
- 真人写的文章会有不完美的停顿和犹豫，AI 写的文章每句话都太"顺"太"漂亮"
- 在适当位置加入犹豫、不确定、坦承不足的表达：
  - "其实当时我们也不确定行不行"
  - "一开始我觉得很玄学"
  - "中间卡了两天，差点放弃"
  - "这个方法对不对，说实话我现在也没完全想清楚"
- 这些不是编造的"假犹豫"——从访谈素材中寻找受访者真实的犹豫和不确定，优先使用

**4. 不能全是正向收益**
- ❌ 全员提效、一路向上、每个人都因 AI 变得更好
- ✅ 主动写出代价和负面：学习成本、调试的崩溃、AI 幻觉带来的返工、老服务改不动的无奈
- 正面和负面的比例建议在 7:3 左右，不是为了"平衡"，是因为读者本能地不信全是好消息

**5. 高频模板句式必须控制**
- 以下句式/词组全文出现**不超过1次**：
  - "这正是……"
  - "这可能是……最大的……"
  - "不是谁替代了谁，而是……"
  - "每个人的边界都在……"
- "第一……第二……第三……第四……"式教科书总结在结尾处**禁止使用**，改用散文化收束

**6. 排比和对称要刻意打破**
- 出现三个以上并列句式时，必须在中间插入一句不同节奏的话来打散
- 章节小标题风格不能完全统一——有的用引语，有的用动作，有的用名字+一个词

### 风格指纹速查卡

| 维度 | 要求 |
|------|------|
| **开头** | 一句话亮矛盾/张力，不铺背景 |
| **结构** | 判断 → 案例 → 更大判断，螺旋上升 |
| **小标题** | 带判断的短句，3-8 个字 |
| **人物** | 靠行动 + 引语立人，不贴标签不写心理 |
| **引语** | 保留口语感，删无效语气词，每句必带信息 |
| **数据** | 嵌入叙事，before vs after 对比 |
| **比喻** | 生活化，少而精，一概念一比喻 |
| **语气** | 冷静克制，态度藏在选材和措辞里 |
| **情绪词** | 不用感叹号，不用空洞夸赞 |
| **段落** | 3-5 句封顶，短长交替 |
| **金句** | 藏在流动中，不加粗不居中 |
| **技术** | 说人话，首次出现解释，之后直接用 |
| **结尾** | 落在"未完成"上，留余味不喊口号 |
| **反模板感** | 章节骨架不重复，金句总结≤2处，注入犹豫感，写出代价，禁教科书式收束 |
| **整体温度** | 冷静的温度——尊重读者智商，但关心文中的人 |

> **一句话口诀：第一句亮矛盾，案例深挖不跳线，引语是弹药数据是锤子，克制但不冷漠，结尾留在路上，全文要有毛边。**

## 疑难排解

**文章风格不对：**
→ 检查写作简报中的文章类型和风格设定是否准确
→ 在素材分析阶段提供参考文章让 agent 学习风格

**素材利用不充分：**
→ 在素材分析阶段检查关键信息提取是否完整
→ 标记素材中"必须使用"的金句或数据

**素材不足怎么办：**
→ 系统会自动触发访谈提纲生成，帮助你补充素材
→ 如无法访谈，系统会基于现有素材继续，标注待补充信息

**文章过长/过短：**
→ 在写作计划中调整各节字数分配
→ 快速模式下可直接指定目标字数

**章节衔接生硬：**
→ 在文章规划阶段特别关注"起承转合"设计
→ 审校阶段重点检查段落间过渡
