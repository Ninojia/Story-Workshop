# 子智能体写作

## 概述

为每个章节派遣独立的子智能体（Subagent），写完后经过两阶段审查，快速迭代。相比批量模式，控制更精细、反馈更及时，适合复杂或高质量要求的文章。

**进入条件：** 已有用户确认的写作计划，且用户选择了"逐节精写"模式
**产出文件：**
- 章节草稿：`docs/writing/drafts/YYYY-MM-DD-<topic>/section-N-<title>.md`
- 完整草稿：`docs/writing/YYYY-MM-DD-<topic>-draft.md`

## 适用场景

| 情况 | 推荐模式 |
|------|---------|
| 文章较复杂，章节间依赖性强 | ✅ 子智能体写作 |
| 高质量要求，需要精细打磨 | ✅ 子智能体写作 |
| 用户想逐节审阅和调整 | ✅ 子智能体写作 |
| 文章结构清晰，可独立写作 | → 用「内容执行」批量模式 |
| 快速出稿，后续再迭代 | → 用「内容执行」快速模式 |

## 执行流程

```
读取写作计划
    ↓
派遣子智能体写第N节 → 子智能体交稿
    ↓
审查1：规格合规 ──不通过──→ 反馈后重新写作
    ↓通过
审查2：内容质量 ──需修改──→ 反馈后修改
    ↓通过
保存章节 → 展示给用户
    ↓
还有更多章节？ ──是──→ 派遣下一个子智能体
    ↓否
合并为完整草稿 → 调用「文章审校」
```

## 子智能体任务简报

为每个章节构建任务简报：

```
撰写第N节「[章节标题]」，文章主题：[Topic]

【写作计划】
读取写作计划：docs/writing/YYYY-MM-DD-<topic>-plan.md
重点关注第N节的所有要求。

【本节要求】
- 定位：[本节在全文中的作用]
- 核心要点：[列表]
- 素材分配：[应使用的数据/引用/细节]
- 目标字数：XXX字
- 写作风格：[风格描述]

【衔接要求】
- 上一节结尾：[引用上一节最后1-2句]
- 本节需从这里自然衔接
- 本节结尾需为第N+1节做铺垫

【写作规范】
- 中文与英文/数字之间加空格
- 每段不超过4句/120字
- 人物引用保持口语化原貌
- 克制使用修饰词
- 数据要有对比

【输出要求】
保存到：docs/writing/drafts/YYYY-MM-DD-<topic>/section-N-<title>.md
包含元数据头（section、title、word_count、status、date）
```

## 两阶段审查

### 审查1：规格合规

对照写作计划检查：

- [ ] 核心要点是否全部覆盖？
- [ ] 计划中分配的素材（数据/引用/细节）是否都使用了？
- [ ] 字数是否在目标的 ±15% 以内？
- [ ] 与上一节的衔接是否自然？
- [ ] 为下一节的铺垫是否到位？

**通过标准：** 所有核心要点覆盖，关键素材使用，字数达标（±15%以内）。注意全文默认约 2000 字，单节不宜过长。

**不通过时的反馈格式：**
```markdown
**审查未通过：规格合规**

缺失项：
- ❌ 未使用分配的数据："小文件从518秒降到8秒"
- ❌ 缺少向第N+1节的过渡
- ⚠️ 字数偏少（目标500字，实际320字）

请补充以上内容后重新提交。
```

### 审查2：内容质量

检查写作质量：

- [ ] 逻辑是否清晰（每段都有明确的表达目的）？
- [ ] 表达是否生动（有场景、有细节、有数据）？
- [ ] 风格是否与全文一致？
- [ ] 句式是否有变化？
- [ ] 有没有空话套话或多余修饰词？
- [ ] 人物引用是否保持了口语化原貌？

**通过标准：** 读起来流畅自然，风格一致，没有明显的表达问题。

**需修改时的反馈格式：**
```markdown
**审查反馈：内容质量需改进**

具体问题：
1. 第2段过于平铺直叙，缺少画面感
   → 建议：加入一个具体的场景细节
2. "效果非常显著" — 空洞
   → 建议：替换为具体数据
3. 结尾过渡太突兀
   → 建议：用一个承上启下的句子

请根据以上反馈修改。
```

## 审查通过后的展示

每节通过后，向用户简要展示：

```markdown
## ✅ 第N节完成：[标题]

**字数：** XXX字（目标：XXX字）
**审查结果：** 规格合规 ✅ | 内容质量 ✅

**本节摘要：** [2-3句话概括本节核心内容]

**已保存到：** docs/writing/drafts/YYYY-MM-DD-<topic>/section-N-<title>.md

---
继续写第N+1节？
```

用户可以在此选择：
1. **继续** — 开始下一节
2. **查看** — 看一下这节的完整内容
3. **修改** — 对这节提出修改意见
4. **暂停** — 先到这里

## 进度追踪

每完成一节后更新进度：

```markdown
## 📊 写作进度：第N/M节

| 状态 | 章节 | 标题 | 字数 |
|------|------|------|------|
| ✅ | 第1节 | [标题] | XXX字 |
| ✅ | 第2节 | [标题] | XXX字 |
| ✅ | 第3节 | [标题] | XXX字 |
| 🔄 | 第4节 | [标题] | 写作中 |
| ⏳ | 第5节 | [标题] | 待开始 |

**进度：** XXX / XXXX 字（XX%）
```

## 处理章节间依赖

当后续章节需要参考前面已完成的章节时：

**子智能体任务简报中额外添加：**
```
【前文依赖】
本节需要参考以下已完成章节：
- 第2节提到的 [具体内容/术语/数据]
- 第3节引入的 [概念/人物/事件]

请确保本节与上述内容保持一致性，并在需要时自然呼应。
```

**具体做法：**
- 每个子智能体获取写作计划 + 所有已完成的章节草稿
- 在任务简报中明确标注依赖关系
- 特别注意术语、人名、数据的一致性

## 全部完成后

所有章节通过审查后：

1. **合并为完整草稿**
   - 按顺序读取所有章节
   - 去除元数据头
   - 合并为单一文档
   - 通读检查整体连贯性

2. **呈现给用户**
   ```markdown
   **全部章节写作完成！**

   | 章节 | 标题 | 字数 |
   |------|------|------|
   | 第1节 | [标题] | XXX字 |
   | ... | ... | ... |
   | **合计** | | **XXXX字** |

   草稿已合并保存到：`docs/writing/YYYY-MM-DD-<topic>-draft.md`

   接下来进入审校环节？
   ```

3. **用户确认后** → 调用 `story-workshop:文章审校`

## 异常处理

| 情况 | 处理方式 |
|------|---------|
| 子智能体连续两次未通过规格审查 | 检查写作计划是否描述清晰，必要时调整计划 |
| 内容质量反复不达标 | 在任务简报中补充风格示例（从已完成的好章节中摘取） |
| 用户对某节不满意 | 根据反馈更新任务简报，重新派遣子智能体 |
| 章节间风格不一致 | 在后续任务简报中附上前文风格样本，要求保持一致 |

## 核心原则

- **独立子智能体 = 独立视角：** 每节由全新的子智能体撰写，避免上下文疲劳
- **两阶段审查 = 质量保障：** 先确保写对了，再确保写好了
- **精细反馈 = 快速迭代：** 每次反馈都要具体到位，不说"写得不够好"，要说"第2段缺少场景细节"
- **上下文传递 = 一致性：** 通过任务简报显式传递前文信息，不依赖子智能体的记忆
- **用户在场 = 方向可控：** 每节完成后给用户审阅的机会，及时纠偏
