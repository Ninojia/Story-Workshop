# 文章规划

## 概述

基于已确认的写作简报，设计完整的文章结构和逐节写作计划。不同文章类型使用不同的结构骨架，确保产出的文章符合该类型的最佳实践。

**进入条件：** 已有用户确认的写作简报（来自素材分析阶段）
**产出文件：** `docs/writing/YYYY-MM-DD-<topic>-plan.md`

## 执行流程

```
读取写作简报 → 选择结构骨架 → 设计章节大纲
    ↓
为每节分配素材和字数 → 设计章节间衔接
    ↓
生成写作计划 → 用户确认？ ──否──→ 调整计划
    ↓是
保存计划 → 选择执行方式
    ↓
批量执行 → 调用「内容执行」
子智能体 → 调用「子智能体写作」
```

## 结构骨架库

### 📖 人物/团队故事骨架

**核心思路：** 以关键场景切入，用时间线或事件线串联，穿插人物对话和细节。

**推荐结构：**
```
1. 🎬 关键场景切入（200-300字）
   用一个具体的、有画面感的场景开头，制造悬念或好奇心

2. 📍 背景铺垫（150-250字）
   交代故事的起因和背景，但不要平铺直叙，要有"为什么值得讲"的张力

3. 🔨 核心叙事×2个板块（每板块300-500字）
   按时间线或主题线展开，每个板块一个核心情节：
   - 板块内要有：事件描述 + 人物反应 + 具体细节/数据
   - 板块间要有：明确的时间跳转或主题转换

4. 📈 成果与高光（200-300字）
   用数据和事实展示成果，穿插情感高光时刻

5. 🎯 回响与余韵（150-200字）
   不要写"总结与展望"，而是：
   - 回扣开头的场景/悬念
   - 或以一个小细节/金句收束
   - 留给读者回味的空间
```

**字数参考：** 约 2000 字（默认）。聚焦 1-2 个核心情节，不铺全时间线。

### 📊 项目复盘骨架

**核心思路：** 干货导向，结构清晰，让读者能快速抓到关键决策和经验。

**推荐结构：**
```
1. 💡 一句话定调（100-150字）
   直接亮出核心结论："我们做了什么，取得了什么结果"

2. 🎯 项目背景（150-250字）
   - 为什么做？解决什么问题？
   - 初始目标是什么？
   - 团队规模和时间线

3. 🔑 关键决策×2-3个（每个250-400字）
   每个决策板块包含：
   - 当时面临的选择
   - 为什么做了这个决定
   - 结果如何（数据支撑）
   - 如果重来会怎么做

4. ⚠️ 踩坑与教训（200-300字）
   - 最大的几个坑是什么
   - 怎么爬出来的
   - 总结为可复用的经验

5. 📊 成果总结（150-250字）
   - 关键指标对比表
   - 超预期的部分 / 未达预期的部分

6. 🔮 可复用的方法论（150-200字）
   - 提炼为3-5条简洁的原则
   - 让其他团队可以直接参考
```

**字数参考：** 约 1800 字（默认）。聚焦 2-3 个关键决策，不面面俱到。

### 🔧 技术分享骨架

**核心思路：** 问题驱动，方案清晰，用类比让技术概念易懂。

**推荐结构：**
```
1. 🤔 问题切入（150-250字）
   - 遇到了什么问题？（用通俗的方式描述）
   - 这个问题为什么重要/困难？
   - 一个让人印象深刻的数据或场景

2. 🔍 现状分析（150-250字）
   - 当前方案的局限是什么
   - 业界常见做法及其不足
   - 理想状态是什么样的

3. 💡 方案设计（300-500字）
   - 核心思路（用一个类比让非技术人员也能理解）
   - 架构/方案要点（可配图）
   - 为什么选这个方案（对比其他候选）

4. 🛠️ 实现细节×1-2个（每个250-400字）
   - 最关键/最巧妙的技术细节
   - 不需要面面俱到，挑最值得分享的
   - 配合代码片段或架构图说明

5. 📈 效果验证（150-250字）
   - 关键指标对比（优化前 vs 优化后）
   - 性能数据、成功率、用户体验提升
   - 用表格或清晰的对比形式呈现

6. 💭 思考与延伸（100-200字）
   - 这个方案的局限和未来改进方向
   - 对类似问题的通用启发
```

**字数参考：** 约 1800 字（默认）。聚焦核心方案，细节用代码/图辅助表达。

### 🚀 产品介绍骨架

**核心思路：** 用户视角切入，突出价值，轻松易读。

**推荐结构：**
```
1. 😩 痛点场景（150-250字）
   - 还原一个用户遇到问题的具体场景
   - 让读者产生"对对对，我也遇到过"的共鸣

2. ✨ 产品/功能亮相（150-200字）
   - 一句话说清楚这个产品/功能是什么
   - 和原来的做法对比："从XX步简化为XX步"

3. 🌟 核心亮点×3个（每个200-300字）
   每个亮点：
   - 用户视角的使用场景
   - 产品方案（怎么实现的）
   - 效果数据或用户反馈

4. 🎯 使用指引（150-250字）
   - 怎么用？简单几步说清楚
   - 适用场景举例
   - 使用入口/获取方式

5. 💬 用户声音/数据验证（100-200字）
   - 真实用户反馈
   - 关键运营数据
   - 一句话收束
```

**字数参考：** 约 1500 字（默认）。聚焦 3 个核心亮点，不贪多。

## 章节设计规范

### 每节必须包含的信息

```markdown
## 第N节：[小标题]（目标：XXX字）

**本节定位：** [这一节在全文中的作用，一句话说清]

**核心要点：**
1. [要点1]
2. [要点2]
3. [要点3]

**素材分配：**
- 使用数据：[具体数据及来源]
- 使用引用：[具体引用及出处]
- 使用场景/细节：[具体描述]

**衔接设计：**
- 承上：[如何从上一节自然过渡]
- 启下：[如何为下一节做铺垫]

**写作提示：**
[针对这一节的特别说明，如语气变化、节奏要求等]
```

### 章节粒度指南

| 文章总字数 | 建议章节数 | 每节字数 |
|-----------|-----------|---------|
| 1000-1500 | 3-4节 | 300-400字 |
| 1500-2000 | 4-5节 | 350-450字 |
| 2000-2500 | 5-6节 | 350-500字 |
| 2500+（需用户确认） | 5-8节 | 400-600字 |

**原则：** 默认按 2000 字规划，每节 350-450 字。每节要能独立表达一个完整的意思，但又不能太长导致读者失去焦点。

## 衔接设计

好文章的秘诀之一是章节间的衔接。以下是常用的衔接手法：

| 衔接手法 | 示例 | 适用场景 |
|---------|------|---------|
| **时间跳转** | "三个月后..." / "倒回2024年底..." | 叙事类 |
| **转折反差** | "但现实很快给出了不同的答案。" | 制造悬念 |
| **问题牵引** | "代码写好了，但怎么验证呢？" | 技术类 |
| **场景切换** | "从会议室回到工位..." | 人物故事 |
| **数据对比** | "改造前是518秒，改造后呢？" | 需要数据冲击力 |
| **主题递进** | "解决了速度问题，更深层的挑战浮出水面。" | 层层推进 |

**规则：** 在写作计划中，为每个章节明确衔接手法，不留给执行阶段即兴发挥。

## 计划确认与执行交接

计划生成后，完整呈现给用户，然后：

**"写作计划已就绪。你可以选择执行方式：**

**1. 批量写作** — 每2-3节为一批，写完给你检查。适合大多数情况。

**2. 逐节精写** — 每节独立完成和审查，适合高质量要求的复杂文章。

**选哪种？"**

**用户选择后：**
- 批量写作 → 调用 `story-workshop:内容执行`
- 逐节精写 → 调用 `story-workshop:子智能体写作`

## 质量检查

保存计划前，确认：
- [ ] 文章结构符合所选类型的骨架
- [ ] 每节有明确的定位和核心要点
- [ ] 素材分配合理，关键数据/引用都有着落
- [ ] 衔接设计到位，章节间不会出现突兀跳转
- [ ] 字数分配合理，加总与目标一致（±10%）
- [ ] 开头足够吸引人（不是从背景介绍开始）
- [ ] 结尾有设计感（不是"总结+展望"）
- [ ] **反模板感检查：** 各章节的叙事骨架是否有差异？把每节的核心结构抽象为一句话，如果超过半数结构相似（如都是"问题→AI解决→提效"），必须重新设计——让不同章节有不同的叙事重心（方法论/态度/失败/工具链/人物性格等）
- [ ] **代价与犹豫分配：** 是否有至少2-3节规划了负面内容（代价、失败、不确定）？全文不能全是正向收益
