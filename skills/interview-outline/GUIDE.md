# 访谈提纲生成

## 概述

当素材分析阶段发现现有素材不足以支撑目标文章类型（尤其是人物故事类）时，自动触发本阶段。基于已有素材（奖项申报材料、项目文档、简报等）智能识别**信息缺口**，生成针对性的访谈提纲，帮助 HR 或采编人员高效完成素材补充。

**典型场景：**
- HR 手里只有奖项申报材料，缺少人物对话、场景细节
- 项目文档详尽但缺少团队成员的个人视角和情感表达
- 产品资料齐全但缺少用户故事和使用反馈

## 触发条件

**自动触发：** 素材分析阶段完成后，检测到以下任一情况：

| 信号 | 说明 |
|------|------|
| 文章类型为「人物/团队故事」但缺少人物直接引语 | 故事类文章没有对话就没有灵魂 |
| 素材中只有事实/数据罗列，缺乏场景描述和细节 | 申报材料典型特征 |
| 写作简报中「素材缺口」项 ≥ 3 条 | 缺口过多，需要系统性补充 |
| 写作简报中「情感高光」和「素材金句」均为空 | 缺乏叙事深度的核心要素 |

**手动触发：** 用户明确要求生成访谈提纲，或说"素材不够""需要做访谈"。

## 执行流程

```
素材分析完成 → 检测素材充分度
    ↓
素材充分？ ──是──→ 跳过，进入文章规划/内容执行
    ↓否
识别信息缺口 → 确定受访对象
    ↓
按受访对象生成差异化问题 → 组装访谈提纲
    ↓
呈现给用户 → 用户确认？ ──否──→ 调整提纲
    ↓是
保存提纲 → 等待用户完成访谈
    ↓
用户提供访谈记录 → 回到素材分析（补充分析）
    ↓
更新写作简报 → 进入文章规划/内容执行
```

## 素材充分度评估

### 评估维度

根据目标文章类型，对以下维度进行打分（1-5分）：

**📖 人物/团队故事所需：**

| 维度 | 1分（严重不足） | 3分（基本够用） | 5分（素材丰富） |
|------|---------------|---------------|---------------|
| 人物对话 | 无任何直接引语 | 有1-2句引用 | 多段原话，口语化、有个性 |
| 场景细节 | 纯抽象描述 | 有时间地点但无画面感 | 有具体场景、动作、环境描写 |
| 情感弧线 | 无任何情感信息 | 有结果但无过程感受 | 有高光、有低谷、有转折 |
| 数据支撑 | 无数据 | 有结果数据 | 有对比数据，有冲击力 |
| 冲突/转折 | 一帆风顺的叙述 | 提到了困难 | 有具体的困境和突破过程 |

**判断规则：**
- 任一维度 ≤ 2分 → 自动触发访谈提纲生成
- 平均分 ≤ 3分 → 建议触发
- 平均分 > 3分 → 素材基本充分，可直接进入下一阶段

**📊 项目复盘所需：**

| 维度 | 不足信号 |
|------|---------|
| 决策过程 | 只有结论没有决策理由和候选方案 |
| 踩坑细节 | 只说"遇到了挑战"没有具体描述 |
| 数据对比 | 只有最终结果没有优化前基线 |
| 方法论 | 有做法但没有提炼为可复用的经验 |

## 信息缺口识别

### 缺口分类

基于已有素材，识别以下类型的信息缺口：

| 缺口类型 | 描述 | 对文章的影响 | 优先级 |
|---------|------|------------|--------|
| **对话缺口** | 缺少人物直接引语和口语化表达 | 文章缺乏真实感和代入感 | 🔴 高 |
| **场景缺口** | 缺少具体的时间、地点、环境、动作描写 | 文章没有画面感 | 🔴 高 |
| **情感缺口** | 缺少人物的内心感受、情绪变化 | 文章没有温度 | 🟡 中 |
| **细节缺口** | 缺少具体的小故事、插曲、意外 | 文章平淡，没有记忆点 | 🟡 中 |
| **数据缺口** | 缺少关键数据或数据缺乏对比基线 | 文章说服力不足 | 🟡 中 |
| **背景缺口** | 缺少项目/事件的来龙去脉 | 读者难以理解上下文 | 🟢 低 |
| **多视角缺口** | 只有单一视角，缺少不同角色的观点 | 文章视角单一，不立体 | 🟢 低 |

### 缺口呈现格式

```markdown
## 素材充分度评估

**目标文章类型：** [类型]
**总体评估：** [素材不足 / 基本够用 / 素材丰富]

### 评分详情

| 维度 | 得分 | 说明 |
|------|------|------|
| 人物对话 | ⭐⭐ | 申报材料中无任何直接引语 |
| 场景细节 | ⭐ | 纯事实罗列，无具体场景 |
| 情感弧线 | ⭐ | 无任何情感信息 |
| 数据支撑 | ⭐⭐⭐⭐ | 有详细的项目数据 |
| 冲突/转折 | ⭐⭐ | 提到"克服困难"但无细节 |

### 关键缺口

🔴 **必须补充：**
1. [缺口描述] — 影响：[对文章的影响]
2. [缺口描述] — 影响：[对文章的影响]

🟡 **建议补充：**
1. [缺口描述] — 影响：[对文章的影响]

**建议：** 需要对 [受访对象] 进行访谈，我来生成访谈提纲？
```

## 受访对象识别

根据素材中的人物信息和文章需求，识别合适的受访对象：

### 受访对象分类

| 角色 | 适用场景 | 访谈重点 |
|------|---------|---------|
| **当事人/主角** | 所有人物故事 | 个人经历、内心感受、关键决策、具体细节 |
| **直属上级** | 人物成长故事、晋升故事 | 评价、关键决策背景、团队视角 |
| **核心同事/搭档** | 团队故事、项目复盘 | 协作细节、侧面印证、补充视角 |
| **下属/团队成员** | 管理者故事 | 领导风格感受、团队氛围 |
| **用户/客户** | 产品故事 | 使用体验、痛点、反馈 |
| **跨部门协作方** | 跨团队项目 | 协作过程、不同视角 |

### 受访对象推荐规则

- **人物故事：** 必访当事人 + 至少1个侧面视角（上级或同事）
- **团队故事：** 核心负责人 + 2-3个不同角色的成员
- **项目复盘：** 项目负责人 + 关键技术骨干
- **产品故事：** 产品负责人 + 典型用户

## 访谈提纲生成

### 问题设计原则

1. **开放式为主：** 多用"能不能讲讲...""当时是什么样的情况"，少用是非题
2. **场景还原：** 引导受访者回到具体场景，"那天的情况还记得吗？"
3. **追问细节：** 准备好追问句，帮 HR 获取更深层的信息
4. **情感挖掘：** 不直接问"你感觉怎么样"，而是通过场景间接触发情感表达
5. **数据核实：** 把需要确认的数据点嵌入问题中
6. **控制数量：** 每位受访者 8-12 个核心问题，不贪多

### 问题类型库

**场景还原类：**
- "XX 那天的情况还记得吗？当时你在做什么？"
- "第一次遇到这个问题是什么时候？当时是怎么发现的？"
- "接到这个任务/决定的那一刻，你的第一反应是什么？"

**决策过程类：**
- "当时为什么选择了这个方案？有没有考虑过其他方案？"
- "做这个决定的时候，最大的顾虑是什么？"
- "回过头来看，如果重新选一次，还会做同样的决定吗？"

**情感挖掘类：**
- "整个过程中，有没有哪个时刻让你印象特别深刻？"
- "最难的时候是什么阶段？怎么熬过来的？"
- "最终看到成果的时候，团队是什么反应？"

**细节追问类：**
- "能具体说说当时的情况吗？比如你们是在办公室还是在会议室讨论的？"
- "那个数据是怎么得出来的？中间有没有什么意外？"
- "你说的 '很多挑战'，能举一个最典型的例子吗？"

**他人视角类（侧面采访用）：**
- "你怎么评价 XX 在这个项目中的表现？"
- "有没有什么事情让你对 XX 的看法发生了变化？"
- "和 XX 合作过程中，有什么让你印象深刻的瞬间？"

## 访谈注意事项

**给 HR/采编人员的建议：**
- 📱 建议录音（征得同意），事后整理比现场记录更完整
- 📝 重点记录：原话金句、具体数字、情感瞬间
- ⏱️ 灵活调整：如果某个话题聊得深，可以适当延长，砍掉不重要的问题
- 🔄 追问比按部就班更重要：受访者说到有意思的点，果断追问
- 🚫 不要引导答案：不说"你是不是觉得很有成就感"，说"当时你什么感觉"

## 提纲呈现与确认

### 呈现规则

1. **先展示素材充分度评估**，让用户理解为什么需要访谈
2. **再展示完整提纲**，包括受访对象、问题和注意事项
3. **标注重点问题**，告知哪些是最关键的（"如果时间有限，至少问这 5 个"）
4. 最后询问：**"提纲可以吗？需要调整问题或增减受访对象吗？"**

### 确认后的流向

**用户确认提纲后：**
1. 保存提纲到 `docs/writing/YYYY-MM-DD-<topic>-interview-outline.md`
2. 告知用户：

```markdown
**访谈提纲已保存到：** `docs/writing/YYYY-MM-DD-<topic>-interview-outline.md`

**接下来：**
1. 按提纲完成访谈
2. 把访谈记录（文字/录音转文字/笔记均可）提供给我
3. 我会补充分析素材，更新写作简报，然后进入正式写作

💡 不需要把所有问题都问完——有价值的素材比覆盖面更重要。
```

**用户提供访谈记录后：**
1. 对新的访谈记录进行素材分析（复用素材分析 skill 的能力）
2. 将新提取的信息合并到已有的写作简报中
3. 重新评估素材充分度
4. 如果充分 → 更新写作简报，进入下一阶段
5. 如果仍有关键缺口 → 告知用户，但不阻塞流程（标注待补充信息）

## 特殊场景处理

### 申报材料转故事

这是最典型的场景。申报材料的特点：
- ✅ 有详细的项目数据和成果
- ✅ 有清晰的时间线
- ❌ 缺少人物个性化表达
- ❌ 缺少具体场景和细节
- ❌ 缺少情感和冲突
- ❌ 语言高度概括化（"攻克难关""全力投入"）

**针对性策略：**
1. 从申报材料中提取**事实骨架**（时间线、关键节点、数据）
2. 识别**需要"填肉"的点**（每个关键节点都需要场景和对话）
3. 生成的问题聚焦于：**还原那些被概括掉的具体场景**
4. 重点追问："申报材料里说'团队攻克了XX难题'——能具体讲讲当时的情况吗？"

### 只需要部分补充

如果素材大部分充足，只缺少个别信息：
- 不生成完整的访谈提纲
- 而是生成 **"3-5 个快速补充问题"**
- 告知用户："不需要正式访谈，花 10 分钟问几个问题就够了"

### 无法访谈的情况

如果用户表示无法进行访谈：
- 不阻塞流程
- 在写作简报的素材缺口中标注清楚
- 在后续写作中：
  - 避免编造对话和细节
  - 侧重于可用的数据和事实
  - 适当调整文章类型建议（如从人物故事调整为项目复盘）

## 核心原则

- **缺口驱动：** 不是为了访谈而访谈，每个问题都对应一个具体的信息缺口
- **问题要具体：** "讲讲你们的项目"是烂问题，"QQ 闪传第一版上线那天发生了什么"是好问题
- **差异化设计：** 不同受访对象看到不同的问题，不是一套问题问所有人
- **实用主义：** 提纲是给 HR 用的，语言要直白，格式要简洁，拿起来就能用
- **追问比问题重要：** 好的追问句比好的问题更重要，要多准备追问方向
- **不贪多：** 8-12 个核心问题足够，信息密度比覆盖面重要
