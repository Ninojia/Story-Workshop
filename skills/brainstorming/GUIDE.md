# 素材分析

## 概述

将用户提供的原始素材（访谈记录、项目文档、产品资料、口述需求等）转化为结构化的写作简报。这是整个写作流程的基石，无论快速模式还是完整模式都不可跳过。

<HARD-GATE>
在用户确认写作简报之前，不得调用文章规划、内容执行或任何写作相关的 skill。即使用户说"直接写"，也必须先完成素材分析并获得确认。
</HARD-GATE>

## 执行流程

```
接收素材 → 素材解读 → 信息提取 → 判断文章类型
    ↓
提出关键问题（最多3个） → 用户补充
    ↓
生成写作简报 → 素材充分度评估
    ↓
素材充分？ ──否──→ 调用「访谈提纲生成」→ 用户完成访谈 → 补充分析 → 更新简报
    ↓是
用户确认简报？ ──否──→ 修改简报
    ↓是
保存简报 → 推荐工作模式
    ↓
快速模式 → 调用「内容执行」
完整模式 → 调用「文章规划」
```

## 素材接收

### 支持的素材类型

| 素材类型 | 处理方式 |
|---------|---------|
| **访谈记录**（.docx/.txt/.md） | 使用 docx skill 读取，提取人物对话、关键事件、数据 |
| **项目文档**（PRD/技术方案） | 提取核心方案、关键指标、技术亮点 |
| **腾讯文档链接** | 通过腾讯文档 MCP 工具获取内容，支持在线文档、表格、幻灯片 |
| **IMA 文档链接** | 通过 IMA 文档 MCP 工具获取内容，支持 IMA 平台的各类文档 |
| **已有文章**（参考风格） | 分析结构、风格、用语习惯，作为写作参考 |
| **口述需求** | 通过对话补全信息，但要高效，不做无意义的问答 |
| **混合素材** | 以上任意组合，交叉验证信息 |

### 素材读取规则

1. **用户提供了文件？** → 主动读取，不要问"要不要我看看这个文件"
2. **工作区有 .docx 文件？** → 使用 docx skill 读取内容
3. **素材是纯文本/markdown？** → 直接读取分析
4. **用户提供了腾讯文档链接？** → 通过腾讯文档 MCP 获取内容（如 MCP 不可用，提示用户手动复制内容或导出文件）
5. **用户提供了 IMA 文档链接？** → 通过 IMA 文档 MCP 获取内容（如 MCP 不可用，提示用户手动复制内容或导出文件）
6. **没有素材文件？** → 进入对话式素材采集（见下方）

### 链接类素材处理

**腾讯文档链接识别：**
- URL 格式：`https://docs.qq.com/...` 或 `https://doc.weixin.qq.com/...`
- 处理流程：识别链接 → 调用腾讯文档 MCP → 获取文档内容 → 按文档类型分析
- MCP 不可用时的降级方案：提示用户"检测到腾讯文档链接，但腾讯文档 MCP 尚未配置。请手动复制文档内容粘贴给我，或导出为 .docx 文件提供给我。"

**IMA 文档链接识别：**
- URL 格式：包含 IMA 平台域名的链接
- 处理流程：识别链接 → 调用 IMA 文档 MCP → 获取文档内容 → 按文档类型分析
- MCP 不可用时的降级方案：提示用户"检测到 IMA 文档链接，但 IMA 文档 MCP 尚未配置。请手动复制文档内容粘贴给我，或导出为文件提供给我。"

## 素材解读与信息提取

### 通用提取清单

从任何素材中，都要提取以下维度的信息（有则提取，无则标记为待补充）：

**人物信息：**
- 核心人物是谁？角色/职位是什么？
- 有哪些值得引用的原话/金句？
- 人物之间的关系和互动？

**事件信息：**
- 关键事件/时间线是什么？
- 有没有冲突、转折、意外？
- 最具戏剧性的片段是什么？

**数据信息：**
- 有哪些关键数据/指标？
- 数据的前后对比（优化前 vs 优化后）？
- 最有冲击力的数据是哪个？

**技术/产品信息：**
- 核心创新点/亮点是什么？
- 关键技术方案是什么？可以用什么类比让外行理解？
- 与竞品或业界做法的差异？

**情感信息：**
- 有没有高光时刻/低谷时刻？
- 团队/个人的情感变化弧线？
- 最打动人的细节是什么？

### 按文章类型的重点提取

**📖 人物/团队故事** → 重点提取：人物金句、关键转折、情感高光、具体场景细节
**📊 项目复盘** → 重点提取：决策过程、数据对比、踩坑经验、可复用方法论
**🔧 技术分享** → 重点提取：问题背景、方案对比、实现细节、效果数据
**🚀 产品介绍** → 重点提取：用户痛点、产品亮点、使用场景、效果数据

## 文章类型判断

根据素材特征和用户意图，判断并推荐文章类型：

| 信号 | 推荐类型 |
|------|---------|
| 素材是访谈记录，有大量对话和个人叙述 | 📖 人物/团队故事 |
| 素材围绕一个已完成的项目，有时间线和结果数据 | 📊 项目复盘 |
| 素材聚焦于技术方案/架构，有实现细节 | 🔧 技术分享 |
| 素材描述一个新产品/新功能，有用户场景 | 🚀 产品介绍 |
| 特征混合 | 询问用户偏好，或推荐最适合的类型并说明理由 |

## 对话式素材采集

当用户没有提供文件素材，或素材信息不足时，通过对话补全关键信息。

### 核心原则

- **最多问3个关键问题**，不做冗长的问答
- **每次只问1个问题**
- **优先选择题**，降低用户回答成本
- **能从素材推断的信息不要问**，展示你的分析能力
- **直奔要点**，不问"你想写什么类型的文章"这种可以自己判断的问题

### 必须确认的信息（按优先级）

1. **核心信息** — 这篇文章最想传达的一件事是什么？（如果素材足够清晰，可以直接提炼让用户确认）
2. **目标读者** — 给谁看的？（内部团队/全公司/外部公开）
3. **特殊要求** — 有没有特定的字数要求（默认约 2000 字）、发布平台要求、需要规避的敏感信息？

### 不需要问的（直接判断）

- 文章类型 → 从素材内容判断
- 写作风格 → 根据文章类型自动适配
- 文章结构 → 在规划阶段处理

## 写作简报

### 简报模板

```markdown
# [主题] — 写作简报

**日期：** YYYY-MM-DD
**文章类型：** [人物故事 / 项目复盘 / 技术分享 / 产品介绍]
**目标读者：** [具体描述]
**目标字数：** [用户指定的字数，未指定则默认约 2000 字]
**写作风格：** [根据类型自动适配的风格描述]

---

## 核心信息
[一句话概括：这篇文章要传递的最核心观点]

## 素材概况

### 关键人物
- [人物1]：[角色]，[关键信息]
- [人物2]：[角色]，[关键信息]

### 关键事件/时间线
1. [事件1] — [时间] — [意义]
2. [事件2] — [时间] — [意义]
3. [事件3] — [时间] — [意义]

### 核心数据
| 指标 | 数值 | 对比/说明 |
|------|------|----------|
| [指标1] | [数值] | [对比] |
| [指标2] | [数值] | [对比] |

### 素材金句
> "[值得直接引用的原话1]" — [说话人]
> "[值得直接引用的原话2]" — [说话人]

### 情感高光
- [最打动人的细节/场景1]
- [最打动人的细节/场景2]

## 写作方向
[2-3句话描述文章的整体走向和叙事策略]

## 建议的文章骨架
1. [开头：用什么场景/事件切入]
2. [主体部分的几个核心板块]
3. [结尾：用什么方式收束]

## 素材缺口
- [如有] 需要补充的信息点
- [如有] 需要核实的数据

## 推荐工作模式
[🚀 快速模式 / 🔬 完整模式]，原因：[简要说明]
```

### 简报呈现规则

1. 生成简报后，**完整呈现给用户**，不要分段确认
2. 特别标注你从素材中的关键发现（"我注意到素材中有个很好的细节..."）
3. 如果有素材缺口，明确告知但不阻塞流程（"建议补充XX信息，但不影响先开始写"）
4. 最后明确询问：**"这个方向可以吗？有需要调整的地方吗？"**

## 简报确认后的流向

**用户确认简报后：**

1. 将简报保存到 `docs/writing/YYYY-MM-DD-<topic>-brief.md`
2. **评估素材充分度：**
   - 如果素材不足（详见「访谈提纲生成」skill 的触发条件）→ 调用 `story-workshop:访谈提纲生成`
   - 如果素材充分 → 根据推荐的工作模式执行：
     - **🚀 快速模式** → 调用 `story-workshop:内容执行`
     - **🔬 完整模式** → 调用 `story-workshop:文章规划`
3. 告知用户即将进入哪个阶段

**如果用户要求修改：**
- 根据反馈调整简报
- 重新呈现修改后的版本
- 再次确认

## 核心原则

- **素材为王** — 好文章的基础是对素材的深度理解，不是模板填充
- **少问多判断** — 展示你对素材的理解力，而不是一味追问
- **信息密度** — 简报要精炼有用，不是面面俱到的流水账
- **尊重原始表达** — 受访者的原话往往比改写更有力量，标记好金句
- **数据要有对比** — "从97%提升到99.9%"比"成功率99.9%"有力得多
